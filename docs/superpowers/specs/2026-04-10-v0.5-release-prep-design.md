# v0.5 Release Prep Design

## Goal

Prepare OpenGGF for the dated `v0.5.20260410` release by baselining `origin/develop` to `master`,
aligning release documentation with the actual v0.5 scope, and letting the existing GitHub release
workflow publish artifacts from `master`.

## Current State

- `origin/master` is an ancestor of `origin/develop`.
- `origin/develop` is the release candidate branch and currently points at `4678db290`.
- Local `master` is stale and should not be used as the source of truth for the release comparison.
- The current working tree contains unrelated modified and untracked files, so release prep should be
  performed in a clean worktree or clean checkout rather than in the active dirty tree.
- `pom.xml` still uses `0.5.prerelease`.
- `README.md` and `CHANGELOG.md` have current v0.5 content, but still label v0.5 as unreleased.
- `ROADMAP.md` still contains old v0.5 exit criteria that were overtaken by later AIZ2 and HCZ work.
- Local release tags differ from remote release tags. The release process must avoid force-updating
  existing tags and should only create the new `v0.5.20260410` tag through the release workflow.

## Release Identity

The release version is `0.5.20260410`.

The expected Git tag is `v0.5.20260410`.

The release date is April 10, 2026.

## Branching and Baseline Strategy

Use a clean release branch from `origin/develop`, named `release/0.5.20260410`.

The release branch should contain only release metadata, documentation, and verification-related
changes. Gameplay and architecture changes are out of scope unless a verification blocker is found.

After verification, update `master` by fast-forwarding it to the release branch tip. Since
`origin/master` is an ancestor of `origin/develop`, this should not require a merge commit. If the
fast-forward check fails, stop and re-evaluate the remote branch state rather than forcing.

## Documentation Scope

Update:

- `pom.xml`: change project version from `0.5.prerelease` to `0.5.20260410`.
- `README.md`: mark v0.5 as released on April 10, 2026.
- `CHANGELOG.md`: mark v0.5 as released on April 10, 2026 and make the analysis range concrete,
  using the release branch tip rather than `HEAD` if possible.
- `ROADMAP.md`: replace stale v0.5 "remaining" language with a current v0.5 retrospective and move
  remaining work into v0.6+ scope.
- User/developer docs that show stale runnable JAR names: either update examples to
  `sonic-engine-0.5.20260410-jar-with-dependencies.jar` or make them version-neutral if the exact
  artifact name is not important.

Do not edit unrelated design plans, historical changelog sections, or disassembly/reference content
unless a release-facing statement is misleading.

## Verification Scope

Before moving `master`, run:

- `git status -sb` in the clean release worktree.
- `git diff --check`.
- `mvn test`.
- `mvn package`.
- A tag existence check for `v0.5.20260410` against `origin`.

If ROM-dependent tests are skipped because local ROM files are unavailable, record that explicitly in
the release notes or final verification summary. Do not claim those tests passed.

## Publish Flow

1. Push the release branch for review or directly fast-forward `master` after local verification,
   depending on the chosen execution workflow.
2. Push `master`.
3. Let `.github/workflows/release.yml` create the GitHub release. The workflow reads the version from
   `pom.xml` and creates tag `v0.5.20260410`.
4. Confirm the workflow completed and the release artifacts exist.
5. After the release is confirmed, decide whether to bump `develop` to `0.6.prerelease` in a separate
   post-release commit.

## Non-Goals

- Do not force-update existing tags.
- Do not rewrite `master` or `develop` history.
- Do not pull unrelated dirty working-tree files into release commits.
- Do not fix non-release gameplay issues unless they block verification.
- Do not publish a manual release tag unless the GitHub workflow cannot be used and the fallback is
  explicitly approved.

## Risks and Mitigations

- **Local tag mismatch:** Avoid `git fetch --tags` for the release prep path. Use `git ls-remote` to
  check the new tag and leave existing tag reconciliation as a separate maintenance task.
- **Dirty active worktree:** Use a clean worktree for release prep and stage exact paths only.
- **Local `master` staleness:** Use `origin/master` for ancestry checks. Refresh branch refs with
  `git fetch origin --no-tags --prune`.
- **Release workflow surprise:** Inspect `.github/workflows/release.yml` before publishing. The
  workflow currently publishes on pushes to `master` and derives the tag from the POM version.
- **Documentation drift:** Search for `0.5`, `0.4.prerelease`, `Unreleased`, `develop`, and
  `v0.5 Remaining` in release-facing docs before final verification.
